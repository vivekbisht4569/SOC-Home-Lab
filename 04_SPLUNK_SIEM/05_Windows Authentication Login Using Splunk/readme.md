# Windows Authentication Investigation using Splunk

##  Objective

The objective of this lab was to investigate Windows authentication events collected in Splunk from the Windows Security Log.

The investigation focused on:

- Verifying available log sources
- Identifying authentication Event IDs
- Investigating successful logins (Event ID 4624)
- Investigating failed logins (Event ID 4625)
- Identifying active accounts
- Understanding Logon Types
- Visualizing login activity over time

---

| Component | Description |
|-----------|-------------|
| SIEM | Splunk Enterprise |
| Endpoint | Windows 11 |

| Investigation | SPL Queries |

# 🎯 Investigation Methodology

This lab follows a SOC analyst investigation workflow instead of memorizing SPL queries.

```
Alert
      ↓
Identify Event Category
      ↓
Identify Log Source
      ↓
Verify Logs Exist
      ↓
Find Relevant Event IDs
      ↓
Investigate Important Fields
      ↓
Correlate Evidence
      ↓
Draw Conclusion
```

**Key Principle**

> A SOC analyst does not begin with an SPL query.

A SOC analyst begins with a question.

The SPL query is only used to answer that question.

---

# Step 1 - Verify Available Log Sources

## Investigation Question

> Which Windows logs are available in Splunk?

### SPL

```spl
index=*
| stats count by source
```

### Result

Available sources discovered:

- WinEventLog:Application
- WinEventLog:System
- WinEventLog:Security
- XmlWinEventLog:Microsoft-Windows-Sysmon/Operational

**Finding**

Confirmed that Windows Security Logs are successfully ingested into Splunk.

---

# Step 2 - Discover Available Event IDs

## Investigation Question

> Which Security Event IDs are present?

### SPL

```spl
source="WinEventLog:Security"
| stats count by EventCode
| sort EventCode
```

### Important Event IDs Found

| Event ID | Description |
|-----------|-------------|
| 4624 | Successful Login |
| 4625 | Failed Login |
| 4634 | Logoff |
| 4648 | Logon with Explicit Credentials |
| 4672 | Special Privileges Assigned |
| 4688 | Process Creation |

**Finding**

Windows Security logging is functioning correctly and contains multiple authentication-related events.

---

# Step 3 - Investigate Successful Logins

## Investigation Question

> Who successfully logged into the machine?

### SPL

```spl
source="WinEventLog:Security" EventCode=4624
| table _time Account_Name Logon_Type Workstation_Name Source_Network_Address
```

### Observations

Accounts observed:

- SYSTEM
- LAPTOP-0VFRID7N$
- viveksinghbisht28@gmail.com
- LOCAL SERVICE
- NETWORK SERVICE
- SplunkForwarder
- Splunkd

```
127.0.0.1
```

Logon Types observed:

```
2
5
7
11
```

**Finding**

Authentication activity was primarily generated by system services and the logged-in user.

# Step 4 - Investigate Failed Logins

## Investigation Question

> Were there any failed login attempts?

### SPL

```spl
source="WinEventLog:Security" EventCode=4625
| table _time Account_Name Logon_Type Workstation_Name Source_Network_Address
```

### Observations

Generated four failed login attempts intentionally.

Source Address:

```
127.0.0.1
```

Logon Type:

```
2
```

**Finding**

Failed logins originated from the local machine and were generated for testing purposes.

No evidence of brute-force activity was observed.

Additional service accounts:

- NETWORK SERVICE
- SplunkForwarder
- Splunkd

**Finding**

---

# Step 6 - Analyze Logon Types

## Investigation Question

> How are users authenticating?

### SPL

```spl
source="WinEventLog:Security" EventCode=4624
| stats count by Logon_Type
```

### Results

| Logon Type | Meaning |
|------------|---------|
| 2 | Interactive Login |
| 5 | Service Login |
| 7 | Workstation Unlock |
| 11 | Cached Interactive Login |

**Finding**

Logon Type 5 was the most common, indicating expected service authentication activity.

---

# Step 7 - Authentication Timeline

## Investigation Question

> When are authentication events occurring?

### SPL

```spl
source="WinEventLog:Security" EventCode=4624
| timechart span=1m count
```

### Observation

Authentication events occurred continuously with no abnormal spikes.

**Finding**

No indication of authentication bursts or brute-force behavior.

# Key Takeaways

This lab reinforced an important SOC principle:

> **SOC analysts do not memorize SPL queries. They ask investigative questions and use SPL to answer them.**

The investigation followed a structured workflow:

1. Identify available logs
2. Discover relevant Event IDs
3. Filter the required event
4. Examine important fields
5. Correlate findings
6. Draw a conclusion

This methodology is reusable for real-world security investigations and forms the foundation of effective SOC analysis.
